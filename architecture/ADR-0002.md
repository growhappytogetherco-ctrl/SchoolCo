# ADR-0002: Option A — Dedicated Guardianship Table for Split-Household Support

**Date:** 2026-06-25
**Status:** Accepted
**Deciders:** Elisa Johnson (Founder), SchoolCo Development

---

## Context

SchoolCo must support families where parents do not live together, may have legal custody
restrictions, and may have different levels of access to their child's records. This is
not a rare edge case — in practice, split households represent a significant portion of
families at any school program. Getting this wrong exposes children to safety risks and
exposes the platform to legal liability.

Two options were evaluated:

**Option A:** Dedicated `guardianships` table — one row per (guardian, student) pair,
with per-field visibility JSON and full custody metadata.

**Option B:** Metadata stored in `organization_members.metadata` JSONB — custody
info stored as a sub-document on the existing membership row.

---

## Decision

**Option A was selected.** A dedicated `guardianships` table with first-class columns
for every custody and visibility concern.

---

## Why Option A

**Security isolation is cleaner.** With Option A, RLS policies on `guardianships`
can guarantee that Parent A never reads Parent B's row — even for the same student.
With Option B, metadata lives on membership rows that the parent must be able to
read for other reasons, making field-level isolation much harder to guarantee cleanly.

**Visibility is per-field, not per-record.** Custody agreements routinely restrict
one parent from seeing health records or incident reports while keeping academics
visible. A dedicated table with `visibility_json` handles this naturally. Achieving
the same in Option B requires complex JSON path queries inside RLS policies, which
are hard to audit.

**Court orders and pickup restrictions need their own columns.** `court_order_on_file`,
`pickup_restrictions`, and `custody_type` are first-class data — they affect physical
safety decisions at pickup time. Burying them in JSONB means staff can't query them
reliably (e.g., `SELECT * FROM guardianships WHERE can_pickup = false`).

**History is preserved correctly.** If custody arrangements change, the old row can be
archived (`archived_at`) and a new one created. With JSONB on membership rows, history
would require custom versioning logic.

**Querying is straightforward.** `SELECT * FROM guardianships WHERE student_id = X`
gives staff the complete picture. With Option B, you'd need JSON extraction across
multiple membership rows.

## Schema Overview

```
students
  id, organization_id, [standard columns]
  first_name, last_name, preferred_name, grade_level
  enrollment_status, enrollment_date, track, homeroom_teacher

guardianships
  id, organization_id, [standard columns]
  profile_id → profiles        ← The guardian
  student_id → students        ← The child
  relationship_type            ← mother/father/grandparent/etc.
  household_label              ← "Johnson Family – Primary"
  custody_type                 ← primary/joint/secondary/supervised/none
  is_legal_guardian            ← boolean
  court_order_on_file          ← boolean (staff-visible only)
  court_order_notes            ← text (staff-visible only)
  is_primary_contact           ← boolean
  is_emergency_contact         ← boolean + order
  can_pickup                   ← boolean
  pickup_restrictions          ← text notes
  visibility_json              ← per-field access flags
  communication_json           ← email/sms/app preferences
```

---

## RLS Rules

| Actor | students | guardianships |
|-------|----------|---------------|
| Parent | Own linked students only (via `is_guardian_of()`) | Own rows only — never another parent's row |
| Volunteer | None | None |
| Teacher | Org students | Org students' guardianships (staff policy) |
| Staff | Org students | Org students' guardianships |
| Registrar | Can insert/update | Can insert/update |
| Admin+ | Full org access | Full org access |

`court_order_notes` — while technically readable by staff at the DB level, the
application layer will strip this field from parent-facing API responses.
A dedicated database view (`guardianships_parent_view`) that excludes this
column will be created in Sprint 1 for clean parent-facing queries.

---

## Key Helper Functions Added

- `is_guardian_of(student_id)` — true if the current user has an active guardianship row
- `can_view_student(student_id, field)` — checks `visibility_json` for a specific field

---

## Consequences

**Positive:**
- Split-household isolation is enforced at the database level — impossible to bypass
- Per-field visibility is queryable and auditable
- Physical safety data (pickup, court orders) has indexed, queryable columns
- Custody history is preserved via soft-archive pattern

**Negative / Trade-offs:**
- More tables to maintain
- Every student data query (attendance, incidents, grades) must join `guardianships`
  and check `can_view_student()` before returning data to a parent session

**Mitigations:**
- `can_view_student()` helper function encapsulates the check in one place
- Will create parent-safe database views in Sprint 1 to simplify application queries

---

## Alternatives Rejected

| Option | Why Rejected |
|--------|-------------|
| Option B: metadata in `organization_members` | Field-level isolation is hard to enforce; custody data buried in JSONB; physical safety data not queryable; history requires custom versioning |
| Single `is_split_household` flag on profiles | Doesn't capture which fields each guardian can see or what their custody arrangement is |
