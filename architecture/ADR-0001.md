# ADR-0001: Multi-Tenant Architecture with Supabase RLS

**Date:** 2026-06-25
**Status:** Accepted
**Deciders:** Elisa Johnson (Founder), SchoolCo Development

---

## Context

SchoolCo must serve multiple organizations simultaneously. Rising Leaders Academy
is the first, but the platform must support additional organizations (foundations,
programs, outreach) without code changes. Each organization needs isolated data,
independent branding, and configurable roles.

---

## Decision

Use **Supabase PostgreSQL with Row Level Security (RLS)** as the multi-tenancy
mechanism. Every table includes `organization_id`. All queries are automatically
scoped to the authenticated user's organization via RLS policies.

**Key design choices:**

1. **Default deny** — No data is accessible unless an explicit RLS policy allows it.
2. **Helper functions** — `is_org_member()`, `has_org_role()`, `is_org_admin()`, etc.
   are defined once in PostgreSQL and reused in all RLS policies. This prevents
   policy drift and makes auditing easier.
3. **9-role hierarchy** — `student_future < parent < volunteer < teacher < staff <
   registrar < admin < full_admin < platform_admin`. Roles are stored as a PostgreSQL
   enum and validated at the database level, not just the application layer.
4. **Single membership table** — `organization_members` links profiles to orgs.
   One user can be `parent` in RLA and `volunteer` in Healing Through Christ Foundation
   with no schema changes.
5. **Active org via localStorage** — In Sprint 0, the selected org is stored in
   `localStorage`. This is a known limitation (see ADR notes). Sprint 1 will migrate
   to a server-side cookie set via a Server Action for SSR compatibility.

---

## Consequences

**Positive:**
- All data isolation is enforced at the database level — impossible to bypass via frontend bugs
- Adding new organizations requires zero code changes
- RLS policies are auditable alongside application code
- Supabase CLI can diff and migrate policies like schema

**Negative / Trade-offs:**
- RLS policies add complexity to queries — must test each role combination
- `localStorage` for active org is not SSR-compatible (see Sprint 1 migration plan)
- Supabase free tier has no read replicas — plan upgrade before launch

**Risks:**
- If a helper function is misconfigured, multiple policies could fail silently
  → **Mitigation:** Verification queries in migration 00002 confirm all functions exist

---

## Alternatives Considered

| Option | Rejected Because |
|--------|-----------------|
| Separate Supabase project per org | Operationally infeasible at scale; can't query across orgs |
| Schema-per-org (PostgreSQL schemas) | Supabase doesn't support dynamic schema creation well |
| Application-layer tenancy only | Relies on developers never forgetting `WHERE org_id = ?`. Too fragile for sensitive student data. |
